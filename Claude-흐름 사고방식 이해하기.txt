상태 5: 98 노드 꺼냄

maxprofit(90) > bound(82, 50) → 남은 노드들 전부 pruning 가능

파란 노드(bound 없음 or bound < 90) → 추가 탐색 의미 없음

BestFS가 DFS보다 빠른 이유 — 직관

DFS: 깊이 우선으로 내려가다 보니 "나쁜 가지"도 끝까지 타고 내려감
BestFS: 항상 "가장 희망적인 번호표"를 가진 노드부터 열어봄
        → 좋은 해를 빨리 발견할수록, 나머지 가지의 pruning 기준이 올라감
        → 결국 탐색 노드 수 자체가 줄어듦

maxprofit이 빨리 올라갈수록 → bound < maxprofit인 노드가 많아짐 → pruning이 많아짐 → 탐색 범위 축소

PQ 관리 패턴

꺼냄(dequeue best bound) 
  → 자식 2개 생성
    → 각 자식의 bound 계산
      → bound > maxprofit 이면 PQ에 넣음
      → bound ≤ maxprofit 이면 버림 (pruning)
  → maxprofit 갱신 (실제 profit이 더 크면)

이게 반복되면서 PQ가 점점 비워지고, 비면 종료.

혼동 포인트: "이동"이 아니라 "경우의 수 나열"

외판원 문제의 상태공간트리는 Dijkstra가 아님.

Dijkstra: 지금 당장 가장 짧은 엣지로 이동 (greedy)
외판원 BestFS: 가능한 모든 경로를 트리로 나열하고, 그 중 최적을 찾음

트리 노드의 의미

1->2 노드

이건 "v1에서 v2로 실제로 이동했다"가 아니라
"v1→v2 경로를 선택한 경우" 라는 분기(branch)야.

그 노드의 자식 {3,4,5} = "다음에 v3, v4, v5 중 어디를 방문할지" 3가지 경우 전부 나열

그럼 뭘 기준으로 트리를 탐색하나?

BestFS → PQ에서 bound가 가장 낮은 노드를 먼저 꺼냄

두 개 다 개념 혼동이 있어. 정리해줄게.

첫 번째: bound의 의미 재정의

bound는 **"추가 이익"이 아니라 "이 노드에서 출발했을 때 얻을 수 있는 총 이익의 상한"**이야.

profit=70인 노드의 bound=98
→ "이 경로로 가면 최대 98까지 얻을 수 있다"
→ 지금까지 번 70 + 앞으로 최대 28 = 98

즉 bound는 누적 포함한 총량이야.

그러면 pruning 조건이 왜 맞냐

bound=50, maxprofit=80

bound=50의 의미: "이 경로로 가봤자 총 이익이 최대 50"

maxprofit=80은 이미 확정된 해.

50 < 80 → 이 경로 끝까지 가도 80을 못 이김 → 볼 필요 없음

"추가로 더 먹을 수 있지 않냐"가 아니라
**"끝까지 가도 이미 찾은 해보다 못하다"**는 거야.

두 번째: 0 노드

맞아. 정확히 봤어.

item을 넣었더니 총 무게가 W 초과
→ 이 노드는 invalid (가방에 안 들어감)
→ profit=0, bound=0으로 처리
→ 0 < maxprofit → 즉시 pruning

무게 초과 = 아예 불가능한 경우라서 bound를 0으로 만들어버려서 자동으로 걸러지게 하는 거야.

Assistant:

5/26/2026 13:41:05

좋은 지적이야. 근데 생각보다 늦지 않아.

BestFS라서 괜찮은 이유

BestFS는 bound가 낮은 것부터 꺼내니까:

처음 몇 번 꺼내다 보면
→ bound가 낮은 노드들이 리프까지 빠르게 도달
→ 초반에 minlength 확정
→ 그 이후부터 pruning 시작

DFS였으면 운 나쁘면 한참 헤매다 리프 도달하지만,
BestFS는 가장 유망한 경로가 제일 먼저 리프에 도달하는 경향이 있어.

그리고 bound 자체가 중간 pruning 역할

리프 전에도 pruning이 아예 없는 건 아니야.

중간 노드에서 bound 계산
→ bound ≥ minlength면 이미 pruning 가능

minlength가 아직 ∞이면 pruning 못 하지만,
첫 번째 리프 도달 후부터는 bound 기반 pruning이 빠르게 작동해.

결론

초반엔 pruning이 약하지만, BestFS 특성상 좋은 해가 빨리 나오고
그 순간부터 pruning이 폭발적으로 증가해서 전체적으로는 효율적이야.

외판원 bound 계산 공식

bound = 현재까지 경로 길이 + 남은 모든 출발 노드의 최소 출발 비용 합

구체적으로

"남은 출발 노드" = 아직 경로에서 출발지로 쓰이지 않은 노드들

각 노드 i마다 "i에서 나가는 엣지 중 최솟값"을 찾아서 다 더함.

단, 최솟값 찾을 때 아래 3가지 제외:

1. i == j                        → 자기 자신으로 가는 경우
2. j가 이미 도착지로 사용됨       → 실제로 갈 수 없는 경로
3. j==1 이고 i==마지막 노드 아님  → 중간에 1로 일찍 귀환하는 경우

슬라이드 예시로 확인

[v1→v2] 경로가 확정된 노드:

경로 길이       = 14
v2 최소 출발   = min{7,8,7} = 7   (이미 출발지로 쓰임 → 제외)
v3 최소 출발   = min{4,7,16} = 4
v4 최소 출발   = min{11,9,2} = 2
v5 최소 출발   = min{18,17,4} = 4

bound = 14 + 7 + 4 + 2 + 4 = 31

v1은 이미 출발지로 쓰였으니 제외, v2도 마찬가지.

핵심 직관

"각 정점에서 반드시 한 번은 나가야 하니까, 각 정점의 최소 출발 비용을 다 더하면 하한이 된다"

실제로는 이것보다 크거나 같을 수밖에 없어서 하한(lower bound) 이 되는 거야.

왜 min값만 골라서 더해도 되냐

bound는 "정확한 경로 길이"가 아니라 "절대 이것보다 작을 수 없는 하한" 이거든.

어떤 경로를 선택하든
각 노드에서 나가는 비용 ≥ 그 노드의 최솟값

그러니까:

실제 경로 길이 ≥ 각 노드 최솟값들의 합

이게 항상 성립해. 도착 노드가 뭔지 몰라도 상관없어.

예시로 보면

v3에서 나가는 비용: {4, 7, 16}
→ 실제로 v3→v4를 선택하든, v3→v5를 선택하든
→ 어떤 선택을 해도 비용은 최소 4 이상

"실제로 어디로 가는지"는 bound 계산에 필요 없어.
그냥 "얼마 이상은 든다"는 하한만 알면 되니까.

그래서 bound의 역할이

정확한 값을 구하는 게 아니라, "이 경로는 아무리 잘해도 X 이상 걸린다"를 빠르게 판단하는 것

minlength보다 이 하한이 이미 크면 → 실제 경로는 더 클 테니 → pruning.

bound의 정체

bound는 "전체 일주여행경로의 하한" 이야. 출발할 때의 하한이 아니라.

왜 각 노드 min값 합이 전체 경로의 하한이 되냐

외판원 문제의 핵심 제약:

모든 노드는 반드시 정확히 한 번 출발지가 된다

그러면:

전체 경로 길이 = v1에서 나가는 비용
              + v2에서 나가는 비용
              + v3에서 나가는 비용
              + v4에서 나가는 비용
              + v5에서 나가는 비용

어떤 경로를 선택하든 이 구조는 변하지 않아.
그리고 각 항은 반드시 해당 노드의 최솟값 이상이므로:

전체 경로 길이 ≥ min(v1 출발) + min(v2 출발) + ... + min(v5 출발)
             = 4 + 7 + 4 + 2 + 4 = 21

출발점 확정 여부는 상관없는 이유

"어느 노드에서 어느 노드로 가는지"는 몰라도:

"어떤 경로든 각 노드에서 최솟값보다 싸게 나갈 수는 없다"

이 사실만으로 하한이 성립해. 출발점이 뭔지, 순서가 뭔지는 bound 계산에 필요 없어.

외판원 문제의 정의 자체야.

외판원 문제 조건

1. 모든 정점 방문
2. 각 정점은 한 번씩만
3. 다시 출발 노드로 돌아옴

이 조건에서 바로 나와.

모든 정점을 한 번씩 방문 = 모든 정점에 한 번씩 도착
다시 출발 노드로 돌아옴 = 출발 노드에도 한 번 도착

그러면 자동으로:

n개 정점 → n개의 엣지 → 각 정점에서 정확히 한 번 출발

순환 구조라서 도착 횟수 = 출발 횟수 = n번, 정점 수도 n개니까 각 정점에서 정확히 한 번 출발은 문제 정의에서 보장돼.
